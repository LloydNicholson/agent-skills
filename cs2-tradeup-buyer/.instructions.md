# CS2 Trade-Up Buyer — Complete Workflow

A portable, end-to-end skill for finding, scouting, and executing profitable CS2 (Counter-Strike 2) marketplace trade-up opportunities. This skill automates the full pipeline: **opportunity discovery → market scouting → float/price validation → user confirmation → automated purchase execution**.

## 🚀 Quick Start

```
/cs2-tradeup-buyer <maxCost> [options]
```

**Examples:**
- `/cs2-tradeup-buyer 20` — Find most profitable trade-up under R20
- `/cs2-tradeup-buyer 50 unique` — Find unique outputs, rank by profit, max R50
- `/cs2-tradeup-buyer 100 second` — Find second most profitable under R100

## What It Does

### 1. **Discover Trade-Ups** 
- Searches for profitable opportunities within your budget
- Filters by profitability, cost, wear conditions
- Returns unique outputs (one best trade-up per item)
- Ranks by profit margin %

### 2. **Scout Steam Market**
- Fetches real-time float data + prices for each input
- Compares against spec (price/float tolerances)
- **Tries all available alternatives** — no limits on substitutes
- Automatically continues with cheaper alternatives if primary skin runs out
- **Rate-limited intelligently**: 2-second delays, anonymous proxy traffic

### 3. **Detect Issues Early**
- ⚠️ Marks items over spec with clear warnings
- 🛑 **Stops immediately on rate limits** — no account damage risk
- Reports shortfalls clearly (e.g., "7 of 10 items found")

### 4. **Confirm & Execute**
- Shows cost breakdown before purchase
- Requires user confirmation 
- Executes purchases with **partial-success handling** — continues even if 1–2 items fail
- Reports success rate (e.g., "9 of 10 purchased")

```
                 │
         Still short? YES
                 │

│       collection/rarity:            │

│ - Collect compliant listings       │
│ - Rate limit hit? STOP, report     │
│                                    │
│ Continue until needed ≥ collected  │
│ or all alts exhausted              │
└────────────────┬────────────────────┘
                  │
┌─ NOTE ─────────────────────────────────┐
│ Each input group and its alternatives │
│ page independently. Only the price    │
│ ceiling can stop a loop. If one input │
│ hits its price ceiling, let the other │
│ inputs continue searching until they  │

│ revisit the blocked input, keep paging │
│ its alternatives as needed.           │
└───────────────────────────────────────┘
┌─ CONFIRM & EXECUTE ────────────────┐

```

## Key Features

| Feature | Details |
|---------|---------|
| **Rate Limiting** | 2-second delays, anonymous proxy, immediate abort on limits |
| **Unauthenticated** | No Steam cookies needed — less risky, faster |
| **Page Size** | 100 listings per query (full market page) |
| **Alternatives** | **Unlimited** — all same-collection/rarity items tried |
| **Partial Success** | Continues even if 1–2 items fail to purchase |

## Parameters

- **`currency`** — Display currency: `ZAR` (default) or `USD`
- **`minProfit`** — Minimum profit % (e.g., `50`, `100`, `150`)

## What Happens When It Runs

### Phase 1: Search (few seconds)
You'll see the MCP tool fetch opportunities from the database, filter by your criteria, and pick the best match.

**Output:**

### Phase 2: Scout Market (2–5 minutes depending on items)
For each input needed, the skill:
1. Waits 2 seconds (rate limit safety)
2. Fetches float data (100 items per query)
3. Collects matching listings (price/float within tolerance)
4. If short, tries alternatives from same collection
5. Reports progress

If every input (including all alternatives) exhausts pages without hitting the goal and the only reason is a price ceiling, deliver a concise summary that lists which inputs remain short, how many pages were scanned per input, which price ceilings blocked progress, and suggest trying a different trade-up or raising the budget. Float filters were respected but never caused the loop to end.

## Rate Limits & Safety

### How Rate Limiting Works
- Float fetches are **unauthenticated** (anonymous proxy IP)
- **2-second delays** between requests prevent quota exhaustion
- Anonymous traffic has **much higher limits** than account-specific traffic

### If Rate Limit Hits
1. ⏸️ **Immediately stops all remaining fetches**
2. 📊 **Reports what was collected** (partial inventory)
3. ❓ **Asks user**: Proceed with shortfall? Or abort?
4. 🚫 **Does NOT retry** — avoids account restrictions

### Safe Practices
- Don't run multiple trade-ups in parallel (serial approach)
- If you hit a rate limit, wait 30+ minutes before trying again
- The 2-second delay was calculated for ~1 hour of sustained scouting

## Common Questions

### Q: Will this hurt my Steam account?
**A:** No. Float fetches are unauthenticated (anonymous proxy), not account-tied. The skill stops immediately on rate limits to prevent any damage.

### Q: How many alternatives are tried?
**A:** **All of them** — no limit. If an input has 20 same-rarity alternatives, all 20 are attempted in price order (cheapest first).

### Q: Can I buy even if one item fails?
**A:** **Yes.** If 9 of 10 items succeed, the trade-up proceeds. Failed items are reported clearly.

### Q: How long does it take?
**A:** Typically 2–5 minutes depending on market availability and how many alternatives are tried. Each fetch waits 2 seconds, so dial that into your expectations.

### Q: What if I can't find enough items?
**A:** The skill reports the shortfall and asks if you want to proceed with fewer inputs. Trade-ups sometimes fail due to market conditions — that's normal.

### Q: What happens if I get a "-32001 Session not found" error?
**A:** This is a transient session timeout. The skill automatically:
1. Reloads itself (creates a new MCP session)
2. Resumes from the last checkpoint (if pagination, picks up where it left off)
3. Retries the operation
4. Logs all recovery steps so you see what's happening

You don't need to do anything — the recovery is automatic and transparent. If recovery fails after 3 attempts, you'll see a clear message with the data collected so far.

### Q: Can I trust auto-recovery after session timeout?
**A:** Yes. The recovery strategy is battle-tested for MCP systems:
- Skill reload creates a fresh session ID
- Checkpoints preserve progress (page numbers, collected items)
- Backoff retries (1s → 2s → 4s) handle transient service glitches
- If service is truly down, you'll be told after 3 failed retries
